# nametochinese · 长尾页 programmatic SEO 打法（执行交接）

> 2026-06-07 战略讨论结论，交给 nametochinese session 执行。
> 来源：研究刘小排（Raphael AI）的"一域名 + 几千长尾页 + 导回工具"打法，确认它特别适合 nametochinese。
> **定位：这是给 nametochinese 持续输血的长期流量引擎（慢线资产，6–12 月爬坡），不是快钱。**

## 一、为什么 nametochinese 是最优试验田
1. **长尾池近乎无穷**：任何词/名字/短语 → 一个"翻译成中文"页。
2. **每页天生独特**（真实翻译结果），躲得过 Google 薄页打击 ← 这是能不能活的命门。
3. **零算力成本**（算法+词库，不烧 API）→ 广告收入近乎纯利。
4. **踩在中文稀缺上**。
5. 已有资产巨大：**~85 万词条 + 33 篇 naming-rules + gov-titles**（AdSense 基建已就绪）。

## 二、模式（一句话）
**一个域名（nametochinese.com）下，用模板 + 现有词库数据，自动批量生成几千~几万个长尾页；每页吃一个搜索词、给一个真实独特结果、并导回主工具；用 sitemap 让 Google 发现；末端用 AdSense 变现。**

## 三、铁律（违反就全站完蛋，重点）
1. **每页必须有真实独特价值**（真翻译 + 含义/拼音/用法/例句），**绝不做空模板**。Google 的 Helpful Content / doorway 打击专杀薄页，重则连累整域降权。
2. **渐进发布，别一次性灌几十万页**。突然涌出海量页 = 垃圾站信号。先放一小簇高价值的，看收录+排名，再逐步放量。
3. **导回工具的入口要自然**（每页一个"用工具翻译你自己的"CTA），但**长尾页本身要先有用**，不是纯跳板。
4. **不堆关键词、不抄袭**。

## 四、第一步：先盘点，别急着生成
让 session 先做（不写新页之前）：
- 现有路由结构、已有多少页、85 万词条以什么结构存的、naming-rules 33 篇覆盖了啥。
- 现有 sitemap.xml 怎么生成的、有没有 AdSense 位预留。
- **目的**：在现有资产上扩，不重复造轮子。

## 五、长尾页类型候选（按"搜索价值 × 我们能独特回答"排，session 需验证搜索量）
1. **英文名 → 中文**：`/[name]-in-chinese`（含中文写法 + 含义 + 拼音 + 念法）—— 搜索量大、情感驱动、天然独特。
2. **常用词/短语 → 中文**：`/how-to-say-[word]-in-chinese`、`/[phrase]-in-chinese`（用 85 万词条，每页真翻译 + 例句）。
3. **含义/主题聚类**：`/chinese-name-meaning-[X]`、按来源/主题归类的 hub 页（也帮内链）。
4. （session 评估）哪些簇搜索量大、竞争弱、我们答得比别人独特 → 先做那一簇。

## 六、技术方向（Next.js）
- 动态路由 + `generateStaticParams` 在构建时批量出页（模板 1 个，数据驱动 N 页）。
- 自动生成 **sitemap.xml**（列全部长尾 URL 交给 Google）+ 长尾页之间**互相内链**（帮爬取 + 留存）。
- 长尾页**不进主导航**（不污染用户体验），只靠 sitemap + 内链 + 搜索落地进入。
- 页面成本≈0（Vercel 托管），域名已有。

## 七、分期路线
- **P1（先验证）**：选 1 个高价值簇（建议"英文名→中文"），生成几百~一两千页，发 sitemap，等收录 + 看 4–8 周排名/流量。
- **P2（看数据再放量）**：P1 真有收录和点击 → 扩到第二簇（常用词/短语），逐步放量。
- **P3**：流量起来后，同一台机器平移到 nameinchi（出口换成 Printful 卖货）。

## 八、变现
- 主：**AdSense**（旅行回来申请，内容量已够过审；长尾页是广告展示的主战场）。
- 辅：长尾页 → 导回工具 → 未来接 nameinchi 起名产品的交叉导流。

## 九、不要做的事
- ❌ 一次性生成几十万页 / 空模板薄页 / 关键词堆砌
- ❌ 把长尾页塞进主导航
- ❌ 指望它快——这是 6–12 月慢线资产，不是 30 天快钱
